# 企微消息采集与会话模型设计 V1

## 1. 文档目的

本文件用于定义企微项目中“企业微信群聊 / 私聊消息接收、记录、组织、分析、反哺”的基础模型，作为后续开发与规划的统一依据。

目标是把客户聊天信息从“原始消息流”转化成“可回看、可分析、可反哺的业务资产”。

---

## 2. 设计目标

本设计希望解决以下问题：

1. 如何记录来自企业微信群和私聊的客户消息？
2. 如何组织消息与会话、客户、业务对象之间的关系？
3. 如何为后续聊天分析和客户方案反哺预留结构？
4. 如何避免项目退化为“只会发消息、不记录和理解客户输入”的系统？

---

## 3. 整体工作流

企微消息相关的最小业务闭环应为：

**消息进入 → 原始消息记录 → 会话归档 → 客户关联 → 内容分析 → 需求/风险/关注点提炼 → 方案更新建议 / 跟进建议**

说明：
- 前四步解决“记下来、找得到”
- 后三步解决“看得懂、用得上”

---

## 4. 核心实体模型

建议至少建立以下六类实体概念。

### 4.1 Message（消息）
表示单条原始聊天消息。

建议字段：
- `message_id`：消息唯一标识
- `source_platform`：来源平台（wecom）
- `chat_type`：会话类型（group / private）
- `conversation_id`：所属会话 ID
- `sender_id`：发送方标识
- `sender_role`：发送方角色（customer / staff / system / bot）
- `sender_name`：发送方名称
- `content_type`：文本/图片/文件/语音/其他
- `content_raw`：原始内容
- `content_text`：可分析文本
- `sent_at`：发送时间
- `received_at`：系统接收时间
- `metadata_json`：原始附加信息
- `linked_customer_id`：关联客户 ID（可为空，待后续关联）
- `analysis_status`：未分析 / 已分析 / 分析失败

作用：
- 保存原始聊天事实
- 为后续分析提供材料

---

### 4.2 Conversation（会话）
表示一个持续的聊天上下文。

建议字段：
- `conversation_id`：会话唯一标识
- `chat_type`：group / private
- `platform_chat_id`：企微侧群或私聊标识
- `conversation_name`：群名或会话展示名
- `customer_scope`：单客户 / 多客户 / 内部协作 / 混合
- `primary_customer_id`：主客户 ID（若能确认）
- `status`：活跃 / 沉默 / 结束 / 待确认
- `started_at`
- `last_message_at`
- `message_count`
- `summary_status`：是否已生成会话摘要

作用：
- 承载消息时间线
- 用于按会话回看与分析

---

### 4.3 ConversationParticipant（会话参与者）
表示一个会话中的参与人。

建议字段：
- `participant_id`
- `conversation_id`
- `user_id`
- `user_name`
- `role_type`：customer / doctor / operator / sales / admin / unknown
- `joined_at`
- `left_at`
- `is_primary_contact`

作用：
- 支持群聊多参与者场景
- 区分客户、我方人员、其他参与者

---

### 4.4 Customer（客户）
表示客户主体。

建议字段（此处仅列与会话有关部分）：
- `customer_id`
- `customer_name`
- `wecom_external_user_id` 或对应外部联系人标识
- `customer_stage`：潜在 / 沟通中 / 已签约 / 服务中 / 风险 / 沉默等
- `latest_need_summary`
- `latest_risk_summary`
- `latest_followup_suggestion`
- `last_interaction_at`

作用：
- 承接会话与业务实体绑定
- 承接分析结果反哺

---

### 4.5 ConversationInsight（会话分析结果）
表示对一段会话或消息集合的结构化分析结果。

建议字段：
- `insight_id`
- `conversation_id`
- `customer_id`（可为空或多值映射）
- `time_window_start`
- `time_window_end`
- `summary_text`
- `need_points_json`
- `concern_points_json`
- `objection_points_json`
- `risk_signals_json`
- `intent_level`
- `stage_judgement`
- `next_action_suggestion`
- `confidence_score`
- `generated_at`
- `generated_by`（system / ai / manual）

作用：
- 把聊天记录转成结构化结论
- 为方案更新和后续动作提供依据

---

### 4.6 CustomerPlanUpdateSuggestion（客户方案更新建议）
表示基于聊天分析给出的方案调整建议。

建议字段：
- `suggestion_id`
- `customer_id`
- `insight_id`
- `suggestion_type`：方案补充 / 风险提醒 / 跟进建议 / 优先级调整 / 需求更新
- `suggestion_text`
- `evidence_refs_json`：引用的消息 / 会话 / insight
- `status`：待确认 / 已采纳 / 已忽略 / 已执行
- `created_at`

作用：
- 将聊天分析正式接入业务动作层

## 5. 消息与会话关系

### 5.1 私聊场景
- 一条私聊消息属于一个 private conversation
- conversation 通常可直接映射到某个客户或某个客户接触窗口
- 消息相对容易与客户直接绑定

### 5.2 群聊场景
- 一条群聊消息属于一个 group conversation
- 一个群可能包含多个参与者，甚至多个客户或我方成员
- 必须通过参与者和上下文进一步做客户归属判断

### 5.3 绑定原则
建议采用三层绑定：

1. **直接绑定**：消息发送者可明确识别为某客户
2. **会话绑定**：消息属于某客户主会话
3. **待确认绑定**：当前无法确定客户归属，需后续分析或人工确认

说明：
- 不要求一开始就 100% 绑定准确
- 先允许消息入库，再逐步补充归属

---

## 6. 消息采集范围建议

### 6.1 当前阶段建议先支持
- 企业微信群文本消息
- 企业微信私聊文本消息
- 基础事件时间线记录

### 6.2 第二阶段再扩展
- 图片
- 文件
- 语音转文本
- 系统事件（入群、退群、标签变更、客户状态事件）

说明：
- 当前应优先拿下最有业务价值的文本主链
- 避免一上来把所有消息类型全做复杂

---

## 7. 最小分析模型

建议当前最小分析维度固定为以下五类：

1. **需求点**
   - 客户明确表达的诉求

2. **关注点**
   - 客户反复提到或重点关心的问题

3. **异议点**
   - 客户犹豫、质疑、拒绝、顾虑的地方

4. **风险信号**
   - 流失风险、负面情绪、延期倾向、推进阻力等

5. **下一步建议**
   - 对我方下一步沟通、方案、动作的建议

说明：
- 最小模型要先确保有用，再逐步做复杂分类

---

## 8. 分析触发机制建议

建议支持三种触发方式：

### 8.1 实时触发
适用于：
- 高风险对话
- 重点客户关键消息
- 明显需求变化

### 8.2 定时汇总触发
适用于：
- 每日会话汇总
- 周期性客户状态更新
- 群聊阶段性总结

### 8.3 人工触发
适用于：
- 指定客户分析
- 指定群聊回顾
- 指定时间窗口总结

---

## 9. 方案反哺路径

分析结果不能只停留在 insight 层，必须接入业务动作。

建议的反哺路径：

**聊天记录 → 会话分析结果 → 客户需求更新 → 客户方案更新建议 → 跟进动作建议 → 后续运营动作**

其中至少要有三个落点：
- 客户当前需求摘要
- 客户当前风险摘要
- 下一步跟进建议

---

## 10. 最小闭环定义

V1 阶段应优先实现的最小闭环为：

1. 接收一条客户群聊或私聊文本消息
2. 将消息存入消息表
3. 将消息归入某个会话
4. 为会话生成一次最小分析结果
5. 生成一条客户方案/跟进建议
6. 可按客户回看消息 + 分析 + 建议

只要这条最小闭环打通，企微项目就会从“只会发消息”进入“开始理解客户互动”的阶段。

---

## 11. 当前实施建议

### 11.1 第一阶段先做什么
- 先把文本消息主链打通
- 先定义最小数据模型
- 先支持私聊和典型群聊场景
- 先跑最小分析结果

### 11.2 当前不要急着做什么
- 不要一开始就追求所有消息类型全覆盖
- 不要一开始就做过重的分析标签体系
- 不要把复杂 CRM 全部一次性塞进来

### 11.3 当前最重要的原则
- 先确保“消息记得住、找得到、能分析、可反哺”
- 再逐步扩复杂度

---

## 12. 文档结论

企微项目 V1 消息采集与会话模型的核心判断是：

> 先以“消息—会话—客户—分析结果—方案建议”五层结构建立最小闭环，
> 再逐步扩展消息类型、分析维度和自动触发能力。

这套模型是后续开发、数据设计、分析设计与业务接入的统一基础。
