# 当前服务器同机独立业务栈部署策略 v1

## 1. 当前结论
企微项目当前不再优先部署到目标服务器 `122.51.79.175`。

改为直接在当前 OpenClaw 所在服务器 `vm-8-15-ubuntu / 150.158.57.114` 上部署，但必须采用“同机部署、独立业务栈”的方式推进。

这意味着：
- OpenClaw / 多Agent 系统继续保留在当前服务器
- 企微项目也部署在当前服务器
- 两者共机，但目录、端口、环境变量、进程/容器、数据库/缓存、反向代理配置都要独立
- 当前阶段目标是先拿到可运行、可联调、可持续迭代的业务环境

---

## 2. 为什么切换策略
旧路线是把企微项目部署到目标服务器 `122.51.79.175`。

但该路线已经暴露出多重问题：
- 多一层跨机管理和跳板复杂度
- 环境不一致，调试链路更长
- 镜像拉取存在明显不稳定因素
- 主Agent 持续参与和排障的效率较低

当前服务器才是日常真实工作环境，也是 OpenClaw 与代码主控环境。把企微项目直接落在当前服务器，更符合“先跑起来、先打通、先可控”的阶段目标。

---

## 3. 当前部署原则

### 3.1 共机但不混跑
企微项目不能直接混入 OpenClaw 现有运行环境中，必须保持栈边界清晰。

### 3.2 优先隔离
至少做到以下独立：
- 项目目录独立
- `.env` 独立
- 服务端口独立
- 容器名/compose project 名独立
- 数据库/缓存实例独立
- Nginx / Caddy / 反代规则独立
- 日志目录独立
- 启停方式独立

### 3.3 优先最小闭环
当前阶段不追求一次性做成最终生产形态，优先拿到：
- backend 可启动
- 数据库可连接
- 演示数据可初始化
- 关键 API 可调用
- 后续可继续迭代

### 3.4 不影响 OpenClaw 现网能力
任何部署动作都不能以牺牲当前 OpenClaw 可用性为代价。

---

## 4. 当前服务器边界认知
基于当前主机检查，已知情况：

### 4.1 当前已监听端口
- `22`：SSH
- `27228`：OpenClaw Gateway（127.0.0.1）
- `27230`：OpenClaw 本地端口（127.0.0.1）

说明：
- 当前没有发现 PostgreSQL `5432`、Redis `6379`、业务服务 `3000` 被宿主机占用
- 企微项目可以优先使用这组端口，但仍建议保留后续重映射空间

### 4.2 当前机风险提示
本机 `docker` 命令当前不可用（PATH 中未找到），说明至少存在以下一种情况：
- Docker 未安装
- Docker 已安装但当前环境 PATH/权限异常
- 当前机后续需要改用非 Docker 首轮启动方案

因此，当前方案不能默认“Docker Compose 一定能直接用”，需要在执行前先确认本机容器能力。

---

## 5. 建议的首轮部署形态
当前建议按“两阶段”推进。

### 阶段 A：先宿主机直跑，完成最小验证
适用于当前 Docker 状态未明的情况。

建议形态：
- PostgreSQL：本机安装或后装容器
- Redis：可暂缓，若首轮未实际使用则不阻塞启动
- Backend：Node.js 直接运行
- 进程守护：先用 `nohup` / `tmux` / `systemd` 中一种，后续再标准化

优点：
- 起步快
- 排障直接
- 不依赖当前 Docker 立即可用

### 阶段 B：再收敛为独立 compose 栈
当当前机 Docker 条件确认无误后，再切到标准 compose 方案：
- `wecom-mvp-postgres`
- `wecom-mvp-redis`
- `wecom-mvp-backend`
- 后续如有前端，再独立加入

这更利于长期维护，但不必阻塞首轮跑通。

## 6. 当前项目的实际落地基础
从代码现状看，项目已经具备最小 backend 启动基础：
- Express 服务入口已完成
- PostgreSQL 连接已接入
- `.env.example` 已有基本字段
- 初始化 SQL 与 seed SQL 已具备
- `/health`、登录、患者、标签、绑定、健康记录、看板等 API 已有初版

这说明当前真正的关键，不是“还没代码”，而是“把当前代码在本机独立跑起来”。

---

## 7. 当前 compose 文件存在的问题
当前 `project/docker-compose.yml` 还是旧草稿形态，主要问题如下：

1. 直接占用宿主机默认端口：
   - `5432:5432`
   - `6379:6379`
   在共机场景下不够稳妥

2. 只定义了 Postgres / Redis，没把 backend 纳入同一套栈

3. 服务名、项目名仍偏通用，不利于与未来其他项目并存

4. 默认镜像源仍是 Docker Hub 官方源，若网络问题存在，后续仍可能复现拉取失败

因此该 compose 文件不应原样当作当前机最终部署方案直接使用。

## 8. 当前推荐的目录与命名约定
建议后续统一采用以下命名口径：

### 8.1 目录
- 工作目录：`/root/.openclaw/workspace/project`
- 若后续要与开发目录分离，可迁到：`/srv/wecom-mvp` 或 `/opt/wecom-mvp`

### 8.2 服务命名
- compose project：`wecom-mvp`
- backend 服务：`wecom-mvp-backend`
- postgres 服务：`wecom-mvp-postgres`
- redis 服务：`wecom-mvp-redis`

当前本轮已收敛：
- compose 草稿服务名已改为 `wecom-mvp-postgres` / `wecom-mvp-redis`
- 不再使用过于泛化的 `postgres` / `redis` 命名

### 8.3 环境文件
- backend 实际运行环境：`project/backend/.env`
- 项目根示例配置：`project/.env.example`
- 若后续拆分运行态说明，统一收口到 `project/ops/`
- 不与 OpenClaw 配置文件混放

当前本轮已收敛：
- `project/.env` 已移出实际运行路径，避免与 `project/backend/.env` 双份并存
- backend 实际运行配置以 `project/backend/.env` 为唯一真实来源

## 9. 当前推荐的端口策略
建议：
- Backend 暂用 `127.0.0.1:3000`
- PostgreSQL 首轮若宿主机直装，可用 `127.0.0.1:5432`
- Redis 首轮若启用，可用 `127.0.0.1:6379`

但如果后续改为容器化并考虑长期共存，建议预留业务侧映射，例如：
- Backend：`127.0.0.1:13000`
- PostgreSQL：`127.0.0.1:15432`
- Redis：`127.0.0.1:16379`

当前本轮已收敛：
- backend 已从 `*:3000` 收回到 `127.0.0.1:3000`
- compose 草稿已改为 loopback 映射：`127.0.0.1:15432:5432`、`127.0.0.1:16379:6379`

原则：
- 宿主机端口对外暴露要克制
- 数据库和 Redis 原则上优先只对本机开放
- 业务 API 如需外部访问，再由反向代理统一接入

---

## 10. 执行优先级
后续推进应按这个顺序：

### P1：确认当前机运行条件
- Node / npm 是否可用
- PostgreSQL 是否已安装或可快速安装
- Docker 是否可用
- 当前机是否已有反向代理

### P2：先跑 backend 最小闭环
- 复制 `.env.example` 为 `.env`
- 调整 DB 连接参数
- 初始化数据库
- `npm install`
- `npm run build`
- `npm run dev` 或守护运行
- 验证 `/health`、登录、核心 API

### P3：再决定是否收敛到 compose
如果 Docker 条件正常，再把 backend + Postgres + Redis 纳入同机独立 compose 栈。

### P4：最后补外部接入
- 域名
- HTTPS
- 企业微信回调地址
- 反向代理转发
