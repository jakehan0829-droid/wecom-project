# 工作台判断层字段清单 + 降噪改写原则 v1

更新时间：2026-03-31

## 1. 目标

为下一阶段“判断层产品化 / 文案降噪”提供统一依据，避免改写过程中再次分散。

本文件只回答两件事：
- 判断层到底保留哪些字段
- 这些字段应该按什么原则改写成业务可读表达

---

## 2. 当前判断层范围

当前判断层包含两块：
- `insight / D4`
- `business-feedback`

它们共同承担的职责是：
- 解释当前为什么要处理
- 说明当前最值得关注的问题
- 给出建议下一步

它们不应该继续承担：
- 原始接口字段展示
- 底层调试信息堆叠
- 与顶部入口摘要重复表达

---

## 3. 建议保留字段清单

### 3.1 insight / D4 建议保留
建议保留：
- `summary`
- `stage`
- `confidence`
- `needs`
- `nextActions`
- `proposalSuggestion`
- `actionSuggestion`

建议弱化或降级：
- `generatedAt`
- `evidenceMessageIds` 条数

说明：
- `generatedAt / evidence 数量` 更像辅助信息，不应抢主表达

### 3.2 business-feedback 建议保留
建议保留：
- `status`
- `summary`
- `needs`
- `risks`
- `nextSteps`

建议弱化：
- 直接暴露原始字段名
- 过多技术性状态词

---

## 4. 判断层改写后的推荐表达结构

推荐收成 4 个问题：

1. **当前判断**
2. **主要风险 / 顾虑**
3. **当前建议动作**
4. **补充依据**

说明：
- 让用户先看到业务结论
- 再看原因与补充信息
- 避免字段平铺式展示

---

## 5. 降噪改写原则

### 原则 1：先业务语言，后字段语言
错误方向：
- 直接展示接口字段名
- 直接展示英文枚举值

正确方向：
- 先把字段解释成业务判断
- 必要时再保留轻量字段信息

### 原则 2：一句话先讲结论
每个判断块优先给：
- 一句业务可读总结

不要一上来就是：
- stage
- confidence
- type
- status

### 原则 3：风险与建议要可执行
不要只说：
- 有风险
- 建议跟进

要尽量说成：
- 风险是什么
- 为什么重要
- 当前建议动作是什么

### 原则 4：弱化调试味信息
以下信息默认降权：
- generatedAt
- evidence 条数
- 内部 suggestionType 名称
- 技术状态词本身

### 原则 5：避免和顶部入口摘要重复
顶部已经回答：
- 这是谁
- 当前状态
- 为什么需要处理
- 建议下一步

判断层就不该再重复同一句话，而应补：
- 为什么这么判断
- 风险/顾虑来自哪里
- 建议动作依据是什么

---

## 6. 推荐改写方向

### insight / D4
从：
- summary
- stage
- confidence
- needs
- nextActions
- proposalSuggestion
- actionSuggestion

改成更像：
- 当前判断
- 当前阶段
- 主要需求 / 顾虑
- 建议方案
- 建议动作
- 辅助可信度信息

### business-feedback
从：
- status
- summary
- needs
- risks
- nextSteps

改成更像：
- 当前业务反馈
- 当前主要风险
- 当前最值得推进的动作
- 最近动作承接情况

---

## 7. 当前阶段结论

下一阶段判断层产品化，建议严格按本文件推进：

- 先定字段保留范围
- 再按降噪原则统一改写
- 最后再做 UI 层表达收口

一句话收口：

**判断层下一阶段的重点，不是增加更多字段，而是把现有字段改写成真正可读、可判断、可执行的业务表达。**
