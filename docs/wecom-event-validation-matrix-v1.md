## 2. 验证状态定义

- **文档支持**：企业微信官方文档存在该类事件/能力描述
- **代码支持**：当前 backend 已有该类事件处理逻辑
- **模拟已验证**：本地 XML / 回放样本已跑通
- **真实已验证**：真实企微环境中已观测到事件进入系统并驱动业务链
- **真实未命中**：已尝试真实触发，但当前未获得有效真实样本
- **未验证**：尚未进入验证阶段

## 5. 当前能力边界判断

### 5.1 已可对内确认的能力

- 企业微信接收消息服务器已正式打通
- `enter_agent` 与文本消息事件已真实验证通过
- 消息/状态/审计/会话状态回写主链已真实验证通过

### 5.2 当前不能对外宣称“已真实打通”的能力

- 外部联系人资料变更事件
- 外部联系人删除事件
- 群聊关闭/解散事件
- 所有 contact/chat 生命周期事件的真实闭环

注意：上述能力并非“代码没有”，而是“当前真实环境尚未验证通过”。

## 6. 当前最合理的后续策略

### 6.1 应继续投入

1. **巩固已真实成功主链**
   - 多条真实文本消息验证
   - `enter_agent` 重复进入验证
   - 幂等与状态稳定性观察

2. **做事件能力边界核对**
   - 对照企业微信文档
   - 明确当前应用在当前业务路径下实际会推送哪些事件
   - 明确 contact/chat 类事件的真实触发前提

3. **围绕真实成功主链继续打磨运营台**
   - 会话头部摘要
   - 状态标签
   - 操作反馈
   - 真消息主链下的动作处理体验

### 6.2 暂不建议继续投入

1. 随机改客户备注继续撞 `edit_external_contact`
2. 随机删客户继续撞 `del_external_contact`
3. 随机关群继续撞 `dismiss_chat`
4. 在未确认事件订阅边界前继续扩更多 contact/chat 事件代码

## 7. 当前阶段结论

当前企业微信项目已经从“本地模拟可跑”推进到“真实消息/进入类事件主链已打通”。

但 contact/chat 生命周期类事件，当前应准确表述为：

**代码支持 + 模拟验证通过，但真实环境尚未验证命中。**

因此当前最合理的推进策略不是继续盲测，而是：

**以真实已成功主链为核心继续产品化，同时做真实事件能力边界核对。**
