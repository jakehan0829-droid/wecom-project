# C 线迁移优先级 v1

更新时间：2026-03-30

## P0：先搬骨架

优先复用这些，不要重做：
- detail 页面整体结构
- 治理入口区
- message 判断区
- audit 区（时间线 / 责任链 / 前后对照）

原因：这些已经在 B 线里验证过工作流闭环，直接决定 C 线起步速度。

---

## P1：再搬候选判断交互

第二优先级复用：
- 候选列表
- top1 / top2 层级
- 推荐摘要
- 当前上下文对照区
- 为什么当前未选 的反向提示

原因：这部分是治理判断效率的核心，但业务字段替换量略大于骨架层。

---

## P2：最后替换业务语义

在 UI 骨架和交互模式复用后，再替换：
- patient -> participant / 群成员 / 映射对象
- matchedBy / patientId / primaryCustomerId 等字段
- patient mapping 动作 -> participant 映射治理动作

原因：业务语义替换应该发生在骨架稳定之后，不应先改语义再重建界面。

---

## 一句话结论

C 线迁移顺序建议是：**先搬骨架，再搬候选判断交互，最后替换业务语义。**
