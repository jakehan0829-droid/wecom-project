# 停旧表双写回滚方案 / 灰度执行卡 v1

更新时间：2026-03-30

## 1. 目标

用于指导 analyze 从“旧表+新表双写”灰度切换到“仅新表写入”时的执行与回滚。

---

## 2. 当前默认前提

当前系统状态：
- analyze：旧表 + 新表双写
- latest insight：优先读新表，查不到回退旧表

灰度目标：
- 暂停旧表写入
- 保留新表写入
- 保留 latest insight 的旧表回退读

也就是说，灰度阶段不是全面删除旧逻辑，而是：

> **先停旧写，不停旧读回退。**

---

## 3. 灰度执行步骤

### Step 1：执行前确认
执行前至少确认：
- 多 conversation 样本验证已完成
- 前端 detail 页展示稳定
- 新表主读链稳定
- 当前版本可快速重新发布

### Step 2：代码切换策略
建议把 analyze 中旧表写入逻辑做成可快速恢复的开关位或明确代码块。

灰度时操作：
- 临时关闭旧表写入
- 保留新表写入
- 不改 latest insight 的旧表回退读

### Step 3：灰度观察
灰度后重点观察：
- analyze 是否仍稳定成功
- latest insight 是否仍稳定返回
- detail 页 insight 区是否正常
- 新表新增记录是否持续产生

---

## 4. 回滚触发条件

出现以下任一情况，建议立即回滚恢复旧表双写：
- analyze 成功率明显下降
- latest insight 出现明显查不到结果
- detail 页 insight 区大面积异常
- 新表写入异常或字段明显缺失
- 发现某条关键链路仍隐性依赖旧表新写结果

---

## 5. 回滚动作

### 快速回滚原则
回滚目标不是回到“旧表主读”，而是：

> **恢复 analyze 的旧表双写能力。**

### 回滚步骤
1. 恢复 analyze 中旧表写入逻辑
2. 重新 build backend
3. 重启 3000 后端服务
4. 重新验证：
   - analyze
   - latest insight
   - detail 页展示

### 回滚后状态
回滚后应恢复为：
- analyze：旧表 + 新表双写
- latest insight：优先读新表，回退旧表

---

## 6. 当前阶段建议

当前已经可以准备灰度执行卡，但仍建议在真正灰度前，至少再补：
- 旧表读取链排查记录
- 灰度窗口内要重点盯的接口/页面名单

---

## 7. 一句话结论

停旧表双写的最稳妥姿势不是“一步删旧逻辑”，而是：

**先灰度停旧写、保留旧读回退、异常时快速恢复旧表双写。**
