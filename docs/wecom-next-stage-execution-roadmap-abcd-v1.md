# 企业微信项目下一阶段执行路线图（按主线 A/B/C/D）v1

更新时间：2026-03-30

## 1. 文档目标

本路线图用于把企业微信项目下一阶段的推进方式，从“按零散任务推进”切换为“按主线推进”。

目标不是简单列 todo，而是明确：
- 每条主线解决什么问题
- 当前处于什么状态
- 下一阶段具体做什么
- 验收标准是什么
- 主线之间如何衔接

---

## 2. 总体判断

当前企业微信项目最合理的推进方式，不是继续散打，而是按四条主线并行但分层推进：

- 主线 A：企微接入与运行稳定化
- 主线 B：映射治理与治理台稳定增强
- 主线 C：participant / 群聊映射治理复用扩展
- 主线 D：消息采集 → 分析 → 反哺 的业务价值主链

其中：
- A/B 是当前已成型主线，需要收稳
- C 是下一阶段可接续扩展线，需要基于 B 复用推进
- D 是项目真正的业务价值主线，必须尽快补成最小闭环

---

## 3. 主线 A：企微接入与运行稳定化

### 3.1 目标
把“真实可跑”推进成“正式稳定运行”。

### 3.2 当前状态
已完成：
- webhook 真实入口打通
- URL 校验通过
- nginx -> backend 真实代理打通
- backend 已切到 `dist/main.js` 正式运行方式
- PM2 / runtime / ops / logs 已补一轮
- enter_agent / 文本消息 / 真实会话推进主链已验证

未完全收口：
- 事件全集真实边界尚未核清
- 正式守护方案、日志路径、部署脚本仍可继续收口

### 3.3 下一阶段任务

#### A1. 收口正式运行方式
- 固定 backend 正式启动方式
- 固定日志路径
- 固定部署/重启脚本
- 固定环境变量管理策略

#### A2. 补运行态巡检与观察
- 明确关键日志检查点
- 固定 smoke test 入口
- 补充最小巡检清单

#### A3. 完成真实事件能力边界核对
- 对照企微官方文档
- 列出：理论支持 / 代码支持 / 模拟已验证 / 真实已验证 / 真实未命中
- 明确哪些事件后续继续投入，哪些暂不投入

### 3.4 验收标准
- backend 运行方式不再依赖临时手工习惯
- 日志与巡检路径明确
- 真实事件能力边界有明确文档说明

---

## 4. 主线 B：映射治理与治理台稳定增强

### 4.1 目标
把当前已形成的映射治理主链，从“阶段性收口成立”推进到“稳定增强期”。

### 4.2 当前状态
已完成：
- 映射稳定化
- unmapped / conflict 观察面
- confirm / reassign / unconfirm / promote-binding
- mapping audit / summary / governance dashboard
- 前端治理台 detail / message / audit / patient 候选区
- `operatorName` 责任链真实联调闭环

当前判断：
- B 线已达到阶段性收口条件
- 已进入稳定增强期

### 4.3 下一阶段任务

#### B1. 完成 audit 区最后一轮统一
- 继续减少最新卡片与历史卡片的结构差异
- 继续统一责任链表达层次
- 保持前后对照块一致性

#### B2. 继续增强候选判断效率
- 优化 top1 / top2 解释逻辑
- 优化“为何当前未选”提示
- 继续增强当前上下文对照表达

#### B3. 固化 B 线汇报与交接材料
- 保持收口结论、决策摘要、一页汇报稿、内部交接稿可用
- 将 B 线明确标记为“稳定增强期”而非“主线未完工”

### 4.4 验收标准
- B 线页面结构不再频繁推翻
- 治理动作、消息判断、责任链展示保持稳定
- B 线具备持续演示、交接、复用条件

---

## 5. 主线 C：participant / 群聊映射治理复用扩展

### 5.1 目标
在不推翻 B 线成果的前提下，把治理能力从 patient 映射治理扩展到 participant / 群聊映射治理。

### 5.2 当前状态
已准备：
- C 线复用迁移清单
- C 线三栏复用清单
- C 线迁移优先级

当前判断：
- C 线不应从零开始
- 应按“先搬骨架，再搬候选判断交互，最后换业务语义”的方式推进

### 5.3 下一阶段任务

#### C1. 先复用 detail 骨架
- 复用治理入口区
- 复用 message 判断区
- 复用 audit 区

#### C2. 再复用候选判断交互
- 复用候选区结构
- 复用 top1 / top2
- 复用推荐摘要、上下文对照、未选原因提示

#### C3. 最后替换业务对象与字段
- patient -> participant / 群成员 / 映射对象
- 替换字段与动作语义
- 建立 participant 治理动作路径

### 5.4 验收标准
- C 线首版页面明显复用 B 线骨架
- 不出现从零重做一套新治理台
- 业务字段替换清晰、动作语义清晰

---

## 6. 主线 D：消息采集 → 分析 → 反哺 的业务价值主链

### 6.1 目标
形成企业微信项目真正的业务价值闭环，而不是止步于接入和治理。

### 6.2 当前状态
当前最薄弱但最关键：
- 消息采集有基础，但未完全形成正式业务主链
- 聊天分析输出结构未完全定死
- insight 生成与回看主链未正式落地
- 方案建议 / 跟进建议路径未打通

### 6.3 下一阶段任务

#### D1. 定义聊天分析输出结构
- 定义最小分析维度：需求点、风险点、异议点、关注点、下一步建议
- 定义结构化字段
- 定义回看方式

#### D2. 明确消息采集最小闭环
- 明确群聊/私聊接入方案
- 明确消息入库字段
- 明确按客户 / 会话 / 时间回看的查询方式

#### D3. 落最小分析主链
- 让一段会话可生成 insight
- 支持 insight 查询 / 回看
- 支持和客户/会话关联

#### D4. 建立反哺路径
- insight -> 客户方案建议
- insight -> 跟进行动建议
- 明确最小业务落点

### 6.4 验收标准
- 至少形成“消息进入 -> 入库 -> 分析 -> 建议”的最小闭环
- 项目开始具备真实业务价值，而不只是治理价值

---

## 7. 执行优先级建议

### P0（立刻进入）
- A3：真实事件能力边界核对
- D1：聊天分析输出结构定义
- D2：消息采集最小闭环设计

### P1（紧随其后）
- A1/A2：运行方式、日志、巡检收口
- D3：最小分析主链落地
- B1：audit 区最后一轮统一

### P2（在主链稳定后推进）
- D4：反哺路径
- C1/C2：C 线骨架与候选判断交互复用
- B2/B3：B 线持续增强与材料固化

### P3（避免过早展开）
- C3：participant 业务语义全面替换
- 更复杂的图表 / 权限 / 全量产品化动作

---

## 8. 推荐推进顺序（最现实版本）

如果下一阶段只能抓最关键的三步，推荐顺序是：

### 第一步
先完成：
- 真实事件能力边界核对
- 聊天分析输出结构定义

### 第二步
再完成：
- 消息采集最小闭环
- 最小分析主链

### 第三步
最后再决定：
- B 线继续增强多少
- 是否正式切 C 线骨架复用

也就是说：

> **先把“能不能形成业务价值闭环”这件事做出来，再决定治理台扩展要走多快。**

---

## 9. 一句话总结

企业微信项目下一阶段最合理的路线是：

**A/B 收稳，D 尽快补成最小业务闭环，C 基于 B 的骨架按优先级复用推进。**
