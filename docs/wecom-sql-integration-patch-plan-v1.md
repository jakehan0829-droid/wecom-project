# 企微 SQL 接入当前 backend 的 patch 方案 V1

## 1. 文档目的

本文件用于明确：企微消息入库与聊天分析相关 SQL 草案，当前应如何接入现有 backend，避免上线方式不清、数据库改动位置不明、运行风险不可控。

---

## 2. 当前判断

基于现有 backend 结构，当前更适合采用：

> **先接入 `project/backend/src/infra/db/init.sql`，作为当前阶段的最小 patch 方案**

原因：
1. 当前项目已有 `init.sql` 作为数据库初始化入口
2. 目前更适合快速推进最小闭环，而不是先引入复杂迁移体系
3. 新表为增量表，不直接破坏现有已运行主表
4. 对当前阶段来说，这是最快进入实现验证的路径

---

## 3. patch 接入建议

## 3.1 当前建议接入位置
- `project/backend/src/infra/db/init.sql`

### 接入内容
- wecom_conversations
- wecom_conversation_participants
- wecom_messages
- wecom_conversation_insights
- 对应索引
- updated_at 触发器函数（如保留）

---

## 3.2 接入原则
1. **增量追加，不覆盖现有结构**
2. **全部使用 IF NOT EXISTS 语义**（当前草案已基本满足）
3. **不修改当前已稳定业务表的含义**
4. **优先保证不会影响现有发送主链**

---

## 3.3 当前不建议先做的事
- 不建议为了这批表先重构整套数据库迁移机制
- 不建议先改动现有核心业务表
- 不建议把接收/分析链与现有发送表强耦合

---

## 4. 风险点

### 风险 1：init.sql 执行时机不一致
需要确认当前 backend 在什么阶段执行 init.sql，避免“文档接上了但真实环境没跑到”。

### 风险 2：真实环境已有部分同名结构
虽然当前草案使用了 IF NOT EXISTS，但仍需确认不存在命名冲突或字段不一致问题。

### 风险 3：后续迁移体系补建时要处理历史结构
当前 patch 方案适合快速推进，但后续若引入更标准迁移体系，需要做好迁移与收口。

---

## 5. 当前推荐策略

### 当前阶段
- 直接将 SQL patch 作为 init.sql 增量部分接入
- 先服务最小消息入库与最小分析闭环

### 后续阶段
- 等企微最小闭环跑通后，再评估是否要把这批表转入正式迁移管理体系

---

## 6. 验收标准

- init.sql 中已增补企微相关表
- 新表不会破坏现有发送主链
- 初始化后数据库具备 message / conversation / insight 所需结构

---

## 7. 文档结论

当前最合适的 SQL 接入策略是：

> 先把企微新增表作为增量 patch 接入 `project/backend/src/infra/db/init.sql`，
> 用最小风险、最快路径支撑消息入库和聊天分析闭环实现；
> 等最小闭环跑通后，再考虑更标准的迁移体系收口。
