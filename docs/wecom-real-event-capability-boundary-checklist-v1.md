# A3 真实事件能力边界核对清单 v1

更新时间：2026-03-30

## 1. 文档目标

本清单用于核对企业微信项目当前“真实事件能力边界”，避免出现以下问题：
- 把代码支持误判为真实已打通
- 把模拟验证误判为真实环境可复现
- 在真实样本不足时继续高风险盲试
- 对外汇报时夸大当前真实能力范围

本清单的目标不是列出所有代码，而是明确：
- 哪些事件理论支持
- 哪些事件代码支持
- 哪些事件模拟已验证
- 哪些事件真实已验证
- 哪些事件真实未命中
- 哪些事件当前不建议继续投入

---

## 2. 核对维度定义

每类事件按以下 5 个维度核对：

### 2.1 理论支持
企业微信官方文档 / 当前应用配置是否理论上支持。

### 2.2 代码支持
项目当前代码中是否已支持解析、落库、状态推进或业务处理。

### 2.3 模拟已验证
是否已通过本地/脚本/构造 payload 验证。

### 2.4 真实已验证
是否已在真实企微环境中收到有效样本并通过主链验证。

### 2.5 真实未命中
代码和模拟都支持，但当前真实环境尚未拿到有效样本。

## 4. 当前可以对外怎么说

### 可以准确表述为
- 企业微信真实接收消息服务器已打通
- URL 校验已通过
- 真实消息/进入类事件主链已验证成功
- 消息、事件状态、自动化审计已可在真实环境落库

### 当前不能准确表述为
- 企业微信所有关键事件均已真实打通
- contact/chat 生命周期事件已在真实环境全部验证
- 真实事件全集已经核清

---

## 5. 当前不建议继续做的事

### 5.1 不建议继续高风险盲试
不建议继续：
- 随机删客户
- 随机改客户信息
- 随机关群
- 仅为了“碰碰运气拿样本”而做真实业务动作

原因：
- 风险高
- 收益低
- 当前已知问题不是代码缺失，而是真实样本与边界前提不明确

### 5.2 不建议继续夸大汇报口径
必须避免：
- 把“模拟已验证”说成“真实已验证”
- 把“代码支持”说成“真实打通”

## 6. 下一步核对动作

### A3-1 对照官方文档核应用事件范围
核对内容：
- 当前企业微信应用类型
- 可接收事件范围
- external contact / 群聊相关事件前提
- 事件触发条件

### A3-2 对照当前服务配置核边界
核对内容：
- 企业微信后台事件配置
- Token / EncodingAESKey / CorpId 对齐情况
- webhook 接入路径
- nginx -> backend 路由

### A3-3 形成“事件能力边界说明”正式文档
目标：
- 输出一份可汇报、可引用、可约束后续投入范围的说明

---

## 7. 当前阶段一句话结论

A3 当前最重要的判断是：

> **真实消息/进入类事件主链已成立，但 contact/chat 生命周期类事件仍属于“代码支持 + 模拟通过 + 真实未命中”的边界待核区。**
