# 旧 insight 表依赖收口评估 v1

更新时间：2026-03-30

## 1. 当前状态

当前 insight 链路已进入双轨阶段：
- analyze：旧表继续写 + 新表同步写
- latest insight：优先读新表，查不到回退旧表

这意味着：
- 新表已经进入主读写链路
- 旧表仍保留兼容职责

---

## 2. 旧表当前还承担什么

### 2.1 analyze 兼容写入
作用：
- 保持历史 insight 相关接口与查询能力不断
- 避免一次性硬切带来回归风险

### 2.2 latest insight 回退读取
作用：
- 防止新表尚未覆盖的旧 conversation 查不到 insight
- 保留平滑切换空间

---

## 3. 是否可以逐步减少旧表依赖

结论：

**可以开始逐步减少，但不建议立刻移除。**

原因：
- 新表已经能稳定承接 latest insight 主读
- D1 关键字段在新表中表达更自然
- `confidence` / `evidenceMessageIds` 在新表中已明显优于旧表兼容逻辑

但当前仍不建议立即停旧表的原因：
- 还未确认是否存在其他隐性调用依赖旧表
- 还未做历史 insight 的覆盖率评估
- 还未完成 detail 页 / 运营台等消费侧的完整切换验证

---

## 4. 推荐收口顺序

### Phase 1：保持旧写 + 新写
当前状态，继续保持。

目标：
- 让新表继续积累真实样本
- 验证消费侧是否稳定依赖新结构

### Phase 2：停止 latest insight 的旧表主依赖
当前已基本做到：
- latest insight 优先读新表
- 旧表仅作为回退兜底

下一步重点：
- 继续观察是否还频繁触发旧表回退

### Phase 3：评估是否停旧表写
当以下条件满足时，可以开始考虑：
- analyze 生成后的最新 insight 都能稳定从新表拿到
- detail 页 insight 展示稳定
- 运营/查询侧不再依赖旧表特有字段

### Phase 4：旧表退化为历史兼容层
最终理想状态：
- 新表为主
- 旧表仅保留历史回看/临时兜底
- 新功能不再向旧表扩展

---

## 5. 当前最合理的判断

当前并不是“旧表已经没用”，而是：

> **旧表已经不适合作为未来主结构，但仍适合暂时承担兼容层职责。**

也就是说：
- 新表已经可以成为主结构
- 旧表不该继续承接新能力设计
- 旧表可以先作为平滑切换期的兜底层

---

## 6. 一句话结论

下一阶段最合理的方向不是“立即删旧表依赖”，而是：

**继续保持新表为主、旧表仅兜底，并逐步停止对旧表新增能力的投资。**
