# 主Agent失联 / token耗尽接管机制 v1

## 目标
当主Agent 因 token 用尽、模型不可用、长时间无响应、任务推进中断等情况失联时，系统仍可由其他 Agent 或人工最小代价接管，不让项目推进完全停住。

---

## 一、失联判定条件
满足以下任一条件，可判定主Agent 已进入“需要接管”状态：

1. **超过一个推进周期无有效更新**
   - 默认阈值：30~60 分钟
   - 且当前任务状态不是“已完成 / 已阻塞待人工”

2. **明确出现 token / quota / model unavailable 类错误**
   - 包括模型额度耗尽、供应商拒绝、API key 失效、模型不可达

3. **主Agent 在关键动作中断后未恢复**
   - 如 build 中断、服务重启中断、联调中断、配置切换中断

4. **巡检失败且无人收口**
   - `runtime-check.js` 返回非 0
   - 且没有新的推进说明或修复动作

---

## 二、接管顺序

### 第一接管人：A1（值班调度）
A1 不负责重度技术修复，但负责：
- 读取 `project/docs/project-progress-board.md`
- 读取本机制文档
- 运行或触发 `project/scripts/runtime-check.js`
- 判断当前属于哪类问题
- 分派给 A2 / A3 / A6 或通知韩聪

### 第二接管层：按问题类型分派

#### A2 接管条件
适用于：
- 代码未完成
- build 失败
- 接口逻辑异常
- 联调结果异常

A2 负责：
- 定位代码问题
- 继续实现或修复
- 补最小验证步骤

#### A3 接管条件
适用于：
- PM2 进程异常
- 服务不可达
- 数据库权限/连接问题
- 环境变量 / 部署 / 运行问题

A3 负责：
- 运行态恢复
- 部署与服务重启
- 数据库连接与权限检查

#### A6 接管条件
适用于：
- 输出已接近完成，但范围或质量不确定
- 是否该收口 / 是否超范围不明确
- 需要对外演示前复核

A6 负责：
- 范围审查
- 风险复核
- 最终收口质量检查

---

## 三、状态板最低要求（必须外置，不依赖上下文）
`project/docs/project-progress-board.md` 至少应持续维护以下信息：

1. 当前阶段
2. 当前主线任务
3. 每个任务的状态（待开始 / 进行中 / 已完成 / 已阻塞待人工）
4. 当前负责人（主Agent / A1 / A2 / A3 / A6）
5. 最近一次有效进展
6. 下一步动作
7. 是否需要韩聪介入
8. 当前最值得关注的 3 件事

没有这些字段时，接管人只能猜，会显著降低接管效率。

---

## 四、最小接管流程

### Step 1：A1 先判断是否真失联或系统是否进入“假绿”
执行：
```bash
node project/scripts/runtime-check.js
pm2 status chronic-disease-backend
pm2 logs chronic-disease-backend --lines 60 --nostream
```
并结合：
- 最近一次汇报时间
- 当前任务状态
- 是否出现 token / quota / model unavailable 报错
- 最近错误日志是否持续出现数据库字段/权限/接口异常

### Step 2：读取状态板
重点看：
- 当前主线任务是什么
- 停在什么步骤
- 下一步动作是什么
- 是否已有明确阻塞

### Step 3：按类型分派
- 代码问题 → A2
- 服务/部署/数据库问题 → A3
- 收口/质量审查问题 → A6
- 外部资源缺失（新 key、企业微信连接、人类确认） → 韩聪

### Step 4：更新状态板
接管人必须更新：
- 当前负责人
- 当前状态
- 最新阻塞点
- 下一步动作
- 是否需要韩聪介入

### Step 5：主Agent 恢复后重新收口
主Agent 恢复后：
- 读取状态板
- 吸收中间接管结果
- 重新统一对外输出

---

## 五、需要韩聪介入的情况
只有以下情况才应升级给韩聪：

1. 需要新的模型 key / base URL / 供应商信息
2. 企业微信真实连接、授权或外部账号操作
3. 高风险动作（生产级删除、覆盖、迁移、暴露）
4. 多个 Agent 都无法继续推进
5. 存在明显成本风险、数据风险、不可逆风险

---

## 六、当前建议的工作边界

### 主Agent
- 总控
- 判断
- 路由
- 收口
- 低风险直接执行

### A1
- 巡检
- 催办
- 失联判定
- 接管调度

### A2
- 代码修复与实现续跑

### A3
- 运行态恢复 / 部署 / 环境 / 数据库

### A6
- 范围与质量复核

---

## 七、v1 限制
当前机制 v1 仍主要依赖：
- 状态板人工维护
- A1 主动巡检或被触发接管
- `runtime-check.js` 的最小健康检查能力

后续可升级方向：
1. 自动定时运行巡检脚本
2. 巡检失败自动通知
3. 状态板结构进一步标准化
4. 接管任务模板化

---

## 八、与多Agent运行制度的衔接
当前接管机制 v1 已开始与以下制度文档衔接使用：
- `docs/multi-agent-duty-operating-card-v1.md`
- `docs/multi-agent-handoff-templates-v1.md`
- `docs/self-evolution-rules-v1.md`
- `docs/project-status-board-template-v1.md`

A1 巡视、A2/A3/A6 接力、主Agent 收口和失败经验沉淀，后续应优先按这些文档继续标准化。

---

## 九、当前结论
v1 目标不是做完整自治系统，而是先消除“主Agent 一断，全盘停摆”的单点故障风险，并逐步推进到可执行的多Agent运行制度。
