# 当前服务器隔离检查 v1

## 结论
按“同机独立业务栈”标准检查后，当前 **还没有发生实质性冲突**，但已经识别出几处 **潜在隔离风险**，需要在真正启动企微项目之前处理。

当前判断：
- **目录隔离：基本成立**
- **端口隔离：当前未冲突，但旧配置有风险**
- **进程隔离：当前成立**
- **环境隔离：当前部分成立，但默认值不适合直接上线**
- **服务入口隔离：当前成立**

---

## 1. 目录隔离

### 现状
- OpenClaw 主工作区：`/root/.openclaw/workspace`
- 企微项目目录：`/root/.openclaw/workspace/project`

### 判断
从目录层面看，企微项目已经位于独立子目录中，未直接混入 OpenClaw 自身代码运行目录结构。

### 结论
**目录隔离基本成立。**

### 风险提示
如果后续把企微项目的 systemd、日志、运行脚本、临时文件直接散落到 workspace 根目录，会削弱隔离性。后续应继续保持：
- 项目配置放 `project/`
- 项目日志放 `project/logs/` 或独立业务日志目录
- 项目守护脚本放 `project/ops/` 或独立目录

---

## 2. 端口隔离

### 当前监听情况
当前已见监听：
- `22`（SSH）
- `127.0.0.1:27228`（OpenClaw Gateway）
- `127.0.0.1:27230`（OpenClaw Local）
- `127.0.0.1:3000`（企微项目 backend）
- `127.0.0.1:5432`（PostgreSQL）

未见监听：
- `6379`

### 判断
就当前宿主机状态而言：
- backend 当前已收敛到 `127.0.0.1:3000`，与 OpenClaw 一样采用 loopback 优先，**当前不冲突且暴露面已收紧**
- PostgreSQL 当前为 `127.0.0.1:5432`，**当前不冲突**
- Redis 若后续启用，仍应优先保持 loopback 或内部网络访问

### 风险点
`project/docker-compose.yml` 当前仍使用：
- `5432:5432`
- `6379:6379`

这在“同机多业务并存”的长期视角下不够稳妥。虽然当前没撞端口，但它会放大未来冲突概率。

另外，backend 代码此前默认监听 `*:3000`，已经在本轮修正为 `127.0.0.1:3000`；这说明后续新增服务也必须显式控制 host，不能默认外放。

### 结论
**当前端口隔离未发生冲突，但旧 compose 配置存在潜在风险。**

未观察到：

- Redis 进程

### 判断
当前不存在“OpenClaw 与企微项目进程混跑”现象。

### 结论
**进程隔离当前成立。**

## 4. 环境隔离

### 当前现状
目前可见的环境文件只有：
- `project/.env.example`

尚未发现：
- `project/.env`
- 项目正式运行环境文件

### 风险点
当前 `.env.example` 中：
- `DB_HOST=postgres`
- `REDIS_HOST=redis`

这说明它默认是为容器网络准备的，不适合当前“宿主机直跑 backend”的首轮方案直接照搬。

若后续在本机直接运行 backend，则应改为：
- `DB_HOST=127.0.0.1`
- `REDIS_HOST=127.0.0.1`（或首轮暂不依赖）

### 结论
**环境隔离当前部分成立，但运行参数尚未切换到当前机独立栈模式。**

## 5. 服务入口隔离

### 当前现状
OpenClaw 当前入口为本机 loopback：
- `127.0.0.1:27228`
- `127.0.0.1:27230`

企微项目当前尚未暴露服务入口。

### 判断
这意味着当前没有和 OpenClaw 入口层冲突。

### 结论
**服务入口隔离当前成立。**

---

## 6. 当前最主要的隔离风险
真正需要处理的不是“已经冲突了”，而是以下三类潜在问题：

### 风险 1：沿用旧 compose 默认端口
会让数据库和 Redis 未来更容易与其他业务相撞。

### 风险 2：沿用容器网络 env 默认值
如果 backend 在宿主机直跑，而 env 仍写 `postgres` / `redis`，启动一定报错。

### 风险 3：后续守护与日志若不单独规划
容易让企微项目运行资产散落到 OpenClaw 主工作区里，弱化隔离。

本轮已先补基础目录：
- `project/logs/`
- `project/ops/`
- `project/runtime/`
用于继续收紧业务运行资产边界。

---

## 7. 当前隔离检查结论汇总

### 已成立
- 目录隔离：成立
- 进程隔离：成立
- 服务入口隔离：成立
- 当前端口无实质冲突：成立

### 待修正
- `.env` 需要切换到当前机直跑模式
- compose 端口策略需要后续优化
- 项目日志/守护方式需要独立规划

---

## 8. 下一步建议
在真正启动企微项目之前，建议先完成这三件事：

1. 生成 `project/.env` 并改为当前机模式
2. 明确 backend 首轮运行方式（宿主机直跑）
3. 先把 PostgreSQL 补齐，再启动 backend

---

## 一句话结论
当前这台服务器上，OpenClaw 与企微项目 **还没有实际打架**；但要真正满足“同机独立业务栈”，必须在启动前把 env、端口策略和后续守护/log 边界再收紧一层。
