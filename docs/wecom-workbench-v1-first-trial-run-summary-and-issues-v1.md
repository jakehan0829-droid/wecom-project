## 4. 本次试跑暴露的问题清单

### 问题 1：试跑入口成本偏高
表现：
- 初始无法直接从台式机访问前端 dev 地址
- 5173 外网不可达
- 3000 初始只监听 127.0.0.1
- 需要额外处理静态托管与监听地址问题后，才能开始试跑

影响：
- 试跑准备成本高
- 不利于后续频繁演示

当前处理：
- 已将前端挂到后端 3000 同域
- 已把后端监听改为 0.0.0.0

后续建议：
- 固化为稳定访问入口，不再依赖临时 dev 端口

### 问题 2：鉴权方式对试跑不够友好
表现：
- 页面需要 Bearer token，但当前无明显引导
- 用户容易误以为是大模型密钥
- 需要额外查后端鉴权方式并手工生成 JWT

影响：
- 初次试跑体验不顺

当前处理：
- 已定位为后端 JWT 鉴权
- 已生成临时 token 完成本次试跑

后续建议：
- 增加试跑说明或 demo 登录方式

### 问题 3：无 pending action 时，执行层容易被误判为“不可用”
表现：
- 当某个会话没有 pending action 时，“动作与反馈”区看起来像点不动
- 用户容易认为是页面坏了，而不是“当前无可执行动作”

影响：
- 执行层可用性认知不稳定

当前处理：
- 通过补一条测试 pending action 完成试跑

后续建议：
- 在空态下明确提示“当前无 pending action，无法执行最小反馈”

### 问题 4：试跑依赖测试数据补齐
表现：
- 当前真实会话不一定天然存在合适的 pending action
- 为了完成闭环，需要人工补一条测试动作

影响：
- 不利于标准化演示

后续建议：
- 预置一条标准试跑样本链路，避免每次临时补数
