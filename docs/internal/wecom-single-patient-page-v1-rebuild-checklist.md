# 单患者页 v1 骨架重构清单

更新时间：2026-03-31

## 1. 重构目标

本次不再继续在旧工作台结构上做局部修补，而是直接按当前已确认的最小主线，重搭单患者页 v1 骨架。

目标页面只服务一条闭环：

> 患者档案 → 本次提炼结果 → 人工修正 / 确认写入

---

## 2. 页面只保留三块主区

### 主区 A：患者档案
用于展示当前稳定档案内容。

保留字段：
- 患者姓名 / 标识
- 当前问题摘要
- 当前状态判断
- 当前需求 / 顾虑
- 当前后续跟进点

### 主区 B：本次提炼结果
用于展示这次互动新增提炼内容。

保留字段：
- 本次互动摘要
- 本次新增问题 / 主诉
- 本次状态变化
- 本次新增需求 / 顾虑
- 本次建议跟进点

### 主区 C：人工修正 / 确认写入
用于承接人工确认后的最终落档内容。

保留内容：
- 可修改字段
- 人工备注
- 确认写入按钮

---

## 3. 页面移除项

本次重构后，以下内容不再出现在单患者页主页面：

- 治理证据链 / 审计联动
- action history
- pending actions
- 试跑导向表达
- 复杂判断包装
- 建议方案 / 建议动作的独立显眼卡片
- 技术状态 / 验证态辅助信息

说明：
- 这些内容不是删除代码能力
- 只是先不在主页面展示

---

## 4. 重构原则

### 原则 1：先骨架，再字段细化
先把三段式骨架搭对，再做字段精修。

### 原则 2：只服务主链
页面上每个区块都必须直接服务：
- 看原档案
- 看本次新增
- 人工确认后写入

### 原则 3：旧工作台逻辑不再做页面中心
旧结构可以保留代码层能力，但不再继续主导页面表达。

### 原则 4：可逆但不犹豫
本次先做页面重构，不碰底层运行主链；如后续需要，可再恢复辅助能力展示，但当前不保留在主页面。

---

## 5. 代码改造顺序

### 第一步
把当前单患者页主页面结构收成三段式。

### 第二步
把旧工作台验证态区块从页面 JSX 中移出主结构。

### 第三步
把原“判断层”字段拆入“患者档案 / 本次提炼结果”。

### 第四步
把“动作与反馈”改成真正的人工修正 / 确认写入区。

### 第五步
完成 build + restart 后实看验证。

---

## 6. 本次完成标准

完成标准不是“页面更漂亮”，而是：

> **用户打开单患者页后，一眼能看懂这页只在做三件事：看当前档案、看本次新增、确认写入。**

---
