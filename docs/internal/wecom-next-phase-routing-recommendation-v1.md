# 企微项目下一阶段路由建议 v1

更新时间：2026-03-30

## 1. 当前判断

当前阶段已经完成：
- 真实 webhook 固化
- 映射治理链闭环
- 治理台接口就绪
- owner 迁移收口材料齐全

因此下一阶段不建议继续无边界扩展当前主题，而应该明确切主线。

---

## 2. 推荐优先级

### 方案 A：真正执行 owner 迁移（最高优先级）
适用条件：
- owner / DBA 可协调
- 有 backend 重启窗口
- 希望尽快把系统从兼容态切到稳定字段 SQL 过滤态

收益：
- 把当前最大现实约束真正解除
- 列表筛选与治理台更接近正式态
- 这条线最有“阶段收官”价值

风险：
- 需要跨人协作
- 需要窗口与权限

推荐度：最高

---

### 方案 B：前端治理台落地（第二优先级）
适用条件：
- owner 窗口暂时拿不到
- 但希望立即把当前接口能力转成可视化页面

收益：
- 快速形成可展示、可操作的治理台
- 可边用边等 owner 迁移窗口

风险：
- 当前仍有兼容态与最终态差异
- 前端需要理解当前空状态 / 枚举字典 / section 口径

推荐度：高

---

### 方案 C：群聊 participant 级映射治理（第三优先级）
适用条件：
- 私聊链路已基本满意
- 需要进一步解决群聊场景的精细归属

收益：
- 能显著提升 group conversation 的业务真实性
- 是映射能力向更复杂场景推进的关键一步

风险：
- 复杂度明显上升
- 容易把当前已收口的主题再次拉散

推荐度：中

---

## 3. 我的默认建议

如果按“最有收官价值”的标准排序：

1. **先推动 owner 真实执行窗口**
2. 如果短期拿不到窗口，再切 **前端治理台落地**
3. 群聊 participant 级映射治理放在这两者之后

---

## 4. 一句话结论

当前下一阶段最值得优先拍板的是：

**先把 owner 迁移从“准备完成”推进到“真实执行完成”；如果现实条件不允许，再把当前已就绪的治理台接口优先落到前端页面。**
