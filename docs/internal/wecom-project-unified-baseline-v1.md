## 2. 项目正式需求基线

### 2.1 项目正式定位

企微项目不是单一页面项目，不是单纯消息发送项目，也不是为了先做一个复杂后台。

企微项目当前更准确的正式定位应定义为：

> **基于企业微信接入模型能力，围绕“群管理机器人 / 群客服”与“个人医生助手”两条核心业务主线，在持续互动中获取信息、完成分析总结，并沉淀完善成员/患者档案的业务系统。**

这里有五个关键点：

1. 基础载体是企业微信
2. 核心增强能力是模型能力接入
3. 核心业务角色之一是群管理机器人 / 群客服
4. 核心业务角色之一是个人医生助手
5. 核心目标是通过互动沉淀档案，而不是先做重后台系统

---

### 2.2 项目两条核心业务主线

#### A. 群管理机器人 / 群客服主线

系统要能够：
- 在群内与群成员互动
- 回答群内常见问题
- 识别群成员的需求、状态与风险信号
- 提炼群成员的关键信息
- 将互动中获得的信息总结后沉淀到成员档案
- 为后续群运营、群管理、重点成员跟进提供依据

这条主线的本质不是“收群消息”，而是：

> **像群管理机器人或群客服一样，在群互动中不断理解成员，并持续完善成员档案。**

#### B. 个人医生助手主线

系统要能够：
- 与患者进行日常互动
- 回答患者问题
- 在持续沟通过程中理解患者状态、需求与顾虑
- 从沟通内容中提炼有价值的患者信息
- 将互动中获得的信息沉淀到患者档案
- 为后续患者服务与跟进提供依据

这条主线的本质不是“维护一个患者列表”，而是：

> **像个人医生助手一样，在持续沟通中理解患者，并持续完善患者档案。**

---

### 2.3 项目核心业务闭环

#### 群管理闭环
群内互动 → 理解成员问题/状态 → 提炼关键信息 → 更新成员档案 → 支撑后续群管理与群运营

#### 患者管理闭环
患者互动 → 理解患者问题/状态 → 提炼关键信息 → 更新患者档案 → 支撑后续服务与跟进

项目核心不是单点问答，而是：
- 互动
- 理解
- 提炼
- 沉淀
- 反哺后续动作

---

### 2.4 模型能力的正式角色

模型能力在项目中的正式角色应定义为：

- 互动理解能力
- 信息提炼能力
- 档案反哺能力
- 业务增强能力

模型的作用不是替代业务人员，而是：
- 提升互动效率
- 降低人工整理成本
- 提高信息沉淀完整度
- 让后续跟进更有依据

---

### 2.5 后台当前阶段定位

后台不是当前阶段的核心产品目标，也不应先做成复杂系统。

后台当前阶段的定位应明确为：

> **档案查看与手动完善的辅助管理面。**

当前后台最小应支持：
- 查看患者档案
- 查看成员档案（如当前阶段纳入）
- 手动补充和修正档案信息
- 为前台互动沉淀结果提供一个可校正的管理面

当前不应优先扩展为：
- 复杂运营后台
- 重型报表后台
- 多角色复杂权限后台
- 大而全后台

---

### 2.6 工作台在项目中的位置

工作台不是整个项目本身，而是当前阶段的重要承接界面。

工作台当前的作用是：

> **把互动上下文、提炼结果、建议动作、执行反馈、历史记录和档案相关信息，收成一个业务人员能看、能接、能修正的承接界面。**

因此：
- 项目总目标仍是“互动 + 提炼 + 档案沉淀”
- 工作台只是当前阶段的重要落地点
- 不能把整个项目误收缩为“工作台项目”

### 3.3 当前阶段认识

当前最明确的阶段结论是：

#### 已经做得较多的
- 企微接入与运行基础
- 发送 / 回写 / 日志基础
- 工作台 V1 承接界面
- 真实试跑 / 标准回归能力
- 判断层与执行层第一轮产品化

#### 还没形成完整主链的
- 群内互动后的成员信息提炼
- 患者互动后的患者信息提炼
- 从互动结果反哺成员/患者档案
- 档案查看与手动完善后台

一句话说：

> **当前最成熟的是“底座 + 承接界面 + 试跑回归”，而最核心的“互动沉淀档案”主线还没有真正完整立起来。**

## 4. 跑偏纠正 / 清理 / 优化原则

### 4.1 当前核心判断

当前项目最核心的问题，不是“没有成果”，而是：

> **承接层做得比核心业务主线更成熟，导致项目重心出现偏移。**

也就是说：
- 底座和承接层先做厚了
- 但核心业务主线还没有真正立起来

---

### 4.2 跑偏判断

#### 跑偏点一：把工作台阶段成果过度上升成项目主目标
纠正原则：
- 工作台回到“承接界面”定位
- 不再把工作台建设等同于项目本体

#### 跑偏点二：为了验证完整性，提前堆厚后台与中间层想象
纠正原则：
- 后台只保留最小可信闭环所需能力
- 暂不继续扩复杂后台和终态级规划

#### 跑偏点三：试跑与验证做得较深，但验证对象偏向承接链，不是核心主线
纠正原则：
- 后续验证重心转向核心业务主线验证
- 不再主要围绕承接界面继续打磨式验证

---

### 4.3 清理 / 冻结建议

#### 建议冻结
- 继续深挖工作台局部文案和 UI 微调
- 继续扩复杂后台结构与聚合接口

#### 建议降级
- 把工作台从“项目中心”降回“阶段载体”

#### 建议清理
- 部分容易制造“系统已很完整”错觉的表达
- 终态感过强的产品化表述
- 过多“更完整 / 更系统 / 更高阶”的规划语言

后续统一改成：
> **是否直接服务核心业务主线**

---

### 4.4 后续优化重点

#### 优化重点一：把主线切回“互动沉淀档案”
优先主线：
- 群互动 → 成员信息提炼 → 成员档案沉淀
- 患者互动 → 患者信息提炼 → 患者档案沉淀

#### 优化重点二：先做最小可信档案闭环
后台先形成：
- 看得到档案
- 看得懂档案
- 能手动补充
- 能手动修正

#### 优化重点三：把验证从“工作台成立”转到“主线成立”
后续验证重点应改成：
- 群成员互动后能否提炼有效信息
- 患者互动后能否提炼有效信息
- 信息能否进入档案
- 档案能否人工校正

#### 优化重点四：收缩后台目标，避免再膨胀
后台后续只围绕：
- 档案查看
- 档案补录
- 档案修正

不再继续扩散到复杂后台系统。

---

## 5. 后续统一判断标准

后续一切新增工作，都必须先回答：

> **它是不是直接服务“群互动 / 患者互动 / 信息提炼 / 档案沉淀”这条主链；如果不是，就先降级、冻结或延后。**

---

## 6. 当前阶段统一结论

当前项目不需要推倒重来。

当前真正要做的是：

1. 承认已有成果真实存在
2. 不再继续把承接层做得更厚
3. 把项目重心重新拉回核心业务主线
4. 用最小可信闭环先把“互动沉淀档案”做出来

一句话收口：

> **清理的不是已有成果，而是错误的重心和过早膨胀的方向；后续统一按“核心业务主线优先”推进。**
