# C 线复用迁移清单 v1

更新时间：2026-03-30

## 目标

如果后续开始推进 C 线（群聊 participant 映射治理），以下 B 线产物建议直接复用，而不是重做。

---

## 一、建议直接复用的模块

### 1. 详情页骨架
- detail 页面结构
- 基本信息区
- 状态区
- 治理入口区
- message 区
- audit 区

### 2. 治理表单能力
- 动作切换
- 字段校验
- operatorName / operatorNote
- 提交后自动刷新联动

### 3. 候选区能力
- 候选列表展示
- top1 / top2 层级
- 匹配强度表达
- 推荐摘要
- 理由分组
- 当前上下文对照区

### 4. message 判断区
- 排序 / 类型 / 条数切换
- 只看可判断消息
- 异常信号摘要
- 建议动作快捷入口
- 被过滤消息展开区

### 5. audit 区
- 最新结果高亮
- 时间线卡片
- 前后对照块
- 历史 audit 列表
- 责任链展示

---

## 二、C 线开始后需要替换/重定义的内容

### 1. 候选对象类型
- B 线候选对象是 patient
- C 线候选对象将变成 participant / 群成员 / 映射对象

### 2. 匹配理由来源
- B 线以 patient、conversation、matchedBy 为主
- C 线应改成 participant 身份、群关系、成员映射线索

### 3. 治理动作语义
- B 线动作围绕 patient mapping
- C 线动作要改为 participant 映射相关动作

---

## 三、迁移建议

### 第一优先级：复用 UI 骨架
先复用：
- 详情页结构
- 候选区结构
- message 判断区结构
- audit 区结构

### 第二优先级：替换业务字段
把 patient / mapping 相关字段替换成 participant / group mapping 相关字段。

### 第三优先级：保留交互模式
尽量复用：
- 候选排序与解释
- message 判断辅助
- audit 责任链与前后对照

---

## 一句话结论

C 线不应从零开始做界面，而应优先复用 B 线已经沉淀出的治理台骨架，再替换业务对象与字段。
