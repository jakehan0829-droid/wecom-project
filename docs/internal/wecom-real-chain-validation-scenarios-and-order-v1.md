# 真实链条验证场景清单 + 验证顺序 v1

更新时间：2026-03-31

## 1. 目标

作为 P0-2“更真实业务链条验证”的统一依据。

目标不是继续验证页面结构，而是开始验证：
- 更真实的业务状态变化
- 判断、动作、反馈、证据之间是否能持续闭环
- 当前链条在哪些真实场景下还能稳定成立

---

## 2. 当前验证重点

P0-2 当前优先验证的不是“更多页面”，而是“更多真实链条场景”。

优先验证：
1. 有建议动作 → 有 pending action → 有反馈回流
2. 有建议动作 → 暂无 pending action
3. 不同状态反馈（done / closed / failed）是否都能正确表现
4. history / 证据链是否能跟着真实变化更新

---

## 3. 真实链条验证场景清单

### 场景 A：标准闭环场景
目标：
- 验证最小链条持续可跑

链路：
- 判断层给出建议
- 执行层存在 pending action
- 提交反馈
- 回流卡片出现
- history 更新

优先级：最高

### 场景 B：有建议但无动作场景
目标：
- 验证判断层与执行层断开时的页面表达是否清楚

链路：
- 判断层仍有建议动作
- 执行层无 pending action
- 页面空态是否能正确解释当前状态

优先级：最高

### 场景 C：不同反馈状态场景
目标：
- 验证 `done / closed / failed` 三种反馈状态是否都能正确影响回流与 history 表达

链路：
- 同一类动作分别以不同状态提交
- 查看回流卡片与 history 区差异

优先级：高

### 场景 D：动作承接一致性场景
目标：
- 验证判断层建议动作与执行层推荐承接动作是否一致

链路：
- 查看判断层建议动作
- 查看执行层高亮推荐动作
- 核对是否语义一致

优先级：高

### 场景 E：证据链同步场景
目标：
- 验证动作与反馈变化后，左侧证据链是否能支撑解释

链路：
- 看消息
- 看最近治理动作
- 看审计联动
- 确认能解释本次处理动作

优先级：中高

---

## 4. 推荐验证顺序

建议按以下顺序推进：

1. 场景 A：标准闭环场景
2. 场景 B：有建议但无动作场景
3. 场景 C：不同反馈状态场景
4. 场景 D：动作承接一致性场景
5. 场景 E：证据链同步场景

原因：
- 先保主链
- 再看断链空态
- 再看状态差异
- 最后看一致性和证据支撑

---

## 5. 当前阶段结论

P0-2 下一步不应再泛泛说“继续验证”，而应严格按场景推进。

一句话收口：

**真实链条验证的关键，不是多试几个页面，而是把不同业务链条场景按顺序跑透。**
