# 工作台 V1 验收清单 + 当前缺口清单 v1

更新时间：2026-03-31

## 1. 当前阶段定位

当前工作台已进入：

> 可演示、可试跑的 V1 结构收口阶段

这一阶段目标不是把所有能力做全，而是先把：
- 页面主次关系
- 判断层 / 执行层 / 证据层分工
- 最小动作反馈闭环
- 真实主链下的操作体验

收成一个稳定可讲、可演示、可继续迭代的 V1 工作台。

---

## 2. V1 目标结构（当前已收口）

### 顶部：入口摘要（总判断）
已完成：
- “这是谁”
- “当前状态”
- “为什么需要处理”
- “建议下一步”

作用：
- 一眼建立处理判断
- 不要求用户先阅读左右两栏大量细节

### 左侧：上下文 / 明细 / 证据区
已完成：
- 基础上下文
  - 会话ID / 会话标识 / 消息数
  - 主客户ID / 当前映射 / 最近动作时间
  - mappingStatus / matchedBy / 会话状态
- 最近治理动作
- 最近消息
- 治理证据链 / 审计联动
- 患者背景信息已并入基础上下文

作用：
- 作为证据与业务背景区
- 不再重复承担顶部摘要角色

### 右侧第一块：深一层判断
已完成：
- insight / D4
- business-feedback
- 判断摘要网格（作为深一层判断，不再与顶部抢总览角色）

作用：
- 承接顶部入口摘要
- 给出更细一层的业务判断和建议

### 右侧第二块：执行层
已完成：
- pending actions
- action history
- 最小反馈表单
- 本次反馈回流

作用：
- 形成“建议 → 执行 → 反馈 → 回流”的最小闭环

## 5. 当前仍存在的缺口

### 缺口 A：判断层内容仍偏“接口直出”
表现：
- insight / business-feedback 虽已分层，但文案表达仍偏接口字段感
- 少量字段对业务人员仍不够自然

影响：
- 演示时能看懂，但还不够产品化

建议：
- 下一阶段补文案归一化与字段降噪

### 缺口 B：执行层结果沉淀仍偏“单次反馈”
表现：
- 已能提交反馈并回流
- 但历史结果的结构化沉淀仍较轻

影响：
- 适合 V1 试跑
- 但还不适合更复杂运营复盘

建议：
- 后续再考虑增强 history 的结构化表达

### 缺口 C：左侧证据链与最近治理动作仍有轻微重叠
表现：
- 最近治理动作与审计联动都在讲“治理发生了什么”
- 但粒度不同，尚未完全统一

影响：
- 当前可接受
- 但后续仍可继续整理

建议：
- 下一阶段决定是否把 recent actions 更明确地降为“摘要证据”，把审计区保留为“最终落库证据”

### 缺口 D：工作台仍缺明确验收场景脚本
表现：
- 页面结构已成型
- 但缺一份“怎么演示 / 怎么试跑 / 怎么验收”的标准脚本

影响：
- 不利于阶段性交付与后续复盘

建议：
- 补一版试跑脚本 / 演示路径

---

## 6. 当前明确不应继续在本阶段深挖的内容

当前阶段不建议继续深挖：
- 新建复杂 feedback 模型
- 新开大聚合接口
- 做重型筛选 / 报表 / 统计页
- 把单个卡片继续微调到过细
- 扩散到 doctor-review 等新支线 UI

原因：
- 当前核心目标是把工作台 V1 收成一个可讲、可试跑的整体
- 不是把局部能力做成产品终态

---

## 7. 当前阶段结论

可以将当前工作台状态表述为：

> **企业微信工作台 V1 已完成第一轮信息架构收口，已具备入口摘要、判断层、执行层、证据层的基本分工，并形成最小动作反馈闭环，可进入试跑与演示准备阶段。**

但也应明确：

> **当前仍属于 V1，可演示、可试跑，不等于正式产品终态。**
