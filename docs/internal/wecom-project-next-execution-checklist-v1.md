# 企业微信项目下一阶段执行清单 v1

更新时间：2026-03-30

## 1. 使用说明

本清单用于承接当前阶段结论，直接指导下一阶段执行。

执行原则：
- 先做真实已成功主链的巩固，不再继续随机盲试事件
- 先确认事件能力边界，再决定是否继续投入 contact/chat 生命周期事件
- 先把运行方式和工作台收口，再考虑更多规则扩展

优先级说明：
- **P0**：必须先做，影响主链可靠性与真实联调稳定性
- **P1**：下一阶段应完成，影响产品化和继续推进效率
- **P2**：可排后，但应进入计划池

---

## 2. P0 清单（必须优先完成）

### P0-1 固化真实 webhook 运行方式
**目标**：把当前临时 3320 运行态收成可稳定复用的正式运行方式。

**任务**：
- 明确 backend 正式监听端口（建议保留 3320 或切换到统一正式端口）
- 明确正式环境变量来源（`.env` / systemd EnvironmentFile / pm2 ecosystem）
- 明确 nginx 正式代理目标
- 明确日志文件路径
- 明确服务重启方式

**验收标准**：
- 服务可通过标准方式启动/停止/重启
- 进程不依赖手工临时命令
- nginx 配置与 backend 监听关系固定
- 日志路径固定且可排查

**风险/备注**：
- 当前 3320 由手工命令启动，稳定性不足
- 若不收口，进程丢失会导致真实回调中断

---

### P0-2 巩固真实消息主链稳定性
**目标**：确认已真实成功的消息/进入类事件不是偶发成功，而是稳定可重复。

**任务**：
- 连续验证多条真实文本消息
- 验证同一真实会话多次消息推进
- 验证 `enter_agent` 重复进入的表现
- 验证 `wecom_messages / wecom_event_state / wecom_automation_audit / wecom_conversations.status` 一致性

**验收标准**：
- 连续多次真实消息均可稳定入库
- 自动化审计稳定落库
- 会话状态推进符合预期
- 无明显重复开单或异常覆盖

**风险/备注**：
- 若主链都不稳定，后续 contact/chat 生命周期扩展没有意义

---

### P0-3 输出真实消息主链验收记录
**目标**：把当前已验证成功的真实能力沉淀成正式验收记录。

**任务**：
- 记录真实成功样本
- 记录真实会话 ID / 状态推进结果
- 记录关键表查询结果
- 更新交付总结与阶段状态文档

**验收标准**：
- 有一份可复查的真实验收记录
- 后续不再依赖聊天记录回忆验证情况

**风险/备注**：
- 若不沉淀，后续会重复做同样验证

---

## 3. P1 清单（下一阶段应完成）

### P1-1 做企业微信事件能力边界核对
**目标**：明确当前应用在真实环境下，到底能收到哪些事件。

**任务**：
- 对照企业微信官方文档
- 对照当前应用类型与配置
- 对照当前真实已验证事件
- 形成“文档支持 / 代码支持 / 模拟已验证 / 真实已验证 / 真实未命中”清单
- 明确 contact/chat 类事件的真实触发前提

**验收标准**：
- 能清楚说明哪些事件值得继续验证
- 能清楚说明哪些事件当前先不投入

**风险/备注**：
- 若不做，会继续陷入随机试备注/删客户/关群的低效循环

---

### P1-2 继续打磨运营台（围绕真实成功主链）
**目标**：把运营台从“最小工作台 v4”推进到“可演示、可试跑”的更稳定版本。

**任务**：
- 会话头部摘要卡
- 操作成功/失败提示
- timeline 颜色标签与高亮
- 当前会话的状态摘要
- 基于真实消息主链的数据体验优化

**验收标准**：
- 非开发角色能看懂当前会话状态
- 常见操作有即时反馈
- 页面演示不依赖解释大量数据库术语

**风险/备注**：
- 当前工作台已可用，但还偏原型态

---

### P1-3 固化当前 nginx / HTTPS / 域名配置
**目标**：把本次真实接入成功路径固化成正式部署资产。

**任务**：
- 记录域名：`www.moshengyuan.com`
- 记录回调 URL：`https://www.moshengyuan.com/api/v1/wecom/webhook`
- 记录 Token / AESKey 使用位置与更新方式
- 记录 nginx 配置路径与关键代理规则
- 记录 certbot / HTTPS 续期方式

**验收标准**：
- 域名与回调配置可复现
- 后续迁移/恢复不再重新摸索

**风险/备注**：
- 当前成功是一步步排出来的，不固化就容易再次走弯路

---

## 4. P2 清单（可排后，但应进入计划池）

### P2-1 重新评估 contact/chat 生命周期事件验证策略
**目标**：在能力边界核对完成后，再决定是否继续验证 `edit_external_contact / del_external_contact / dismiss_chat`。

**任务**：
- 基于文档与应用能力判断是否值得继续投入
- 若值得，设计最小真实样本验证方案
- 若不值得，明确改成“模拟保证 + 文档边界说明”处理

**验收标准**：
- 不再靠随机业务动作撞事件
- 每次验证前都有明确触发假设

---

### P2-2 守护进程化 / 部署脚本化
**目标**：把当前 backend 启动方式从手工命令推进到正式化。

**任务**：
- 选择 systemd 或 pm2
- 写标准启动配置
- 固定日志目录
- 固定重启命令

**验收标准**：
- 可一键重启
- 可一键查看状态
- 回调不中断风险显著降低

---

### P2-3 更新交付包与项目状态板
**目标**：把这轮真实联调进展同步到项目文档体系。

**任务**：
- 更新交付总结
- 更新项目状态板
- 更新演示/联调说明
- 更新测试资产文档

**验收标准**：
- 项目状态与真实进展一致
- 后续对内对齐不再靠口头说明

---

## 5. 当前不建议做的事情

- 随机改客户备注继续撞 `edit_external_contact`
- 随机删客户继续撞 `del_external_contact`
- 随机关群继续撞 `dismiss_chat`
- 在真实边界未确认前继续扩大量 contact/chat 事件代码
- 把“模拟已过”当成“真实已过”对外表达
