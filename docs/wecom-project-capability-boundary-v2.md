# 企微项目能力边界 V2

## 1. 文档目的

本文件用于正式定义企业微信项目的能力边界、目标范围、当前阶段重点与非目标，避免项目被错误理解为“单纯消息发送项目”。

V2 版本明确：企微项目是一个**基于企业微信的客户互动、触达、记录、分析、方案反哺与业务推进系统**。

---

## 2. 项目总目标

企微项目的目标不是只完成企业微信接入，也不是只完成消息发送。

项目总目标是：

> 建设一个能够基于企业微信群聊与私聊承接客户互动、沉淀会话记录、分析客户需求与风险、反哺客户方案与后续动作，并可稳定运行的业务系统。

项目至少覆盖六类能力：

1. **接入能力**
2. **触达能力**
3. **记录能力**
4. **分析能力**
5. **反哺能力**
6. **运行能力**

---

## 3. 六大能力边界

### 3.1 接入能力

目标：打通企业微信应用、域名、服务、接口与运行基础。

包含内容：
- 企业微信应用接入
- 可信域名校验
- 后端服务启动与运行
- 基础 API 通路
- 与当前服务器部署环境兼容
- 与 OpenClaw 共机部署但边界独立

当前状态：
- 已基本打通并进入稳定化阶段

完成标准：
- 服务可持续运行
- 接入链路可复用
- 运行边界清晰

---

### 3.2 触达能力

目标：支持系统向客户发起主动触达与互动。

包含内容：
- 企业微信消息发送
- 动作发起与状态回写
- 发送结果记录
- 发送统计与基础观察面
- 后续跟进动作的执行基础

当前状态：
- 已完成主链路打通，正在继续增强

完成标准：
- 可真实发送
- 可回看发送结果
- 可按动作查询
- 失败可基本定位

---

### 3.3 记录能力

目标：支持从企业微信群和私聊中采集并沉淀客户聊天信息与互动记录。

包含内容：
- 群聊消息接收
- 私聊消息接收
- 消息原始内容记录
- 会话维度归档
- 客户与消息关联
- 互动事件沉淀
- 会话历史回看能力

当前状态：
- 目前尚未形成正式落地主链，是 V2 必须纳入的核心能力

完成标准：
- 至少能够稳定记录客户群聊和私聊消息
- 可按客户、会话、时间回看消息
- 不再依赖手工翻聊天记录

---

### 3.4 分析能力

目标：把聊天信息转化为结构化业务判断。

包含内容：
- 客户意图识别
- 需求提炼
- 关注点提炼
- 异议识别
- 风险信号识别
- 阶段状态判断
- 跟进建议生成

当前状态：
- 尚未形成正式能力链，是 V2 的重点补齐项

完成标准：
- 系统能够从聊天中提炼结构化分析结论
- 分析结果可查询、可回看、可用于业务动作

---

### 3.5 反哺能力

目标：让聊天分析结果真正用于完善客户方案与需求判断。

包含内容：
- 客户需求理解更新
- 客户画像更新
- 方案调整建议
- 跟进策略建议
- 优先级变化判断
- 风险升级与处理建议

当前状态：
- 目前尚未显式落地主链，但必须成为企微项目核心价值的一部分

完成标准：
- 聊天分析结果能进入客户方案与业务动作
- 系统不只“看懂”，还能“推动下一步”

---

### 3.6 运行能力

目标：保证整个企微项目可部署、可维护、可观察、可排障、可持续运行。

包含内容：
- 配置边界清晰
- 运行态文件归位
- 运维文档清晰
- 日志与状态可查
- 观察面持续增强
- 巡检与异常处理能力
- 自动触发与定时汇报能力（逐步补齐）

当前状态：
- 已完成一轮关键收口，仍需持续增强

完成标准：
- 项目不依赖个人记忆运转
- 出现故障能快速定位
- 日常巡检与交接成本可控

---

## 4. 项目主线重新定义

V2 起，企微项目必须同时沿两条主线推进：

### 主线 A：企微触达与运行稳定化
包含：
- 接入
- 发送
- delivery log
- dashboard
- 运行边界
- 排障能力

### 主线 B：客户会话记录与分析反哺
包含：
- 群聊/私聊消息接收
- 会话记录
- 内容分析
- 客户需求提炼
- 方案更新建议
- 跟进动作建议

说明：
- 主线 A 决定项目“能不能稳定运行和触达客户”
- 主线 B 决定项目“能不能真正服务业务与方案优化”

若只推进主线 A，项目会退化成“消息发送系统”；
只有 A+B 同时推进，项目才符合真实业务预期。

---

## 5. 当前阶段重点

当前阶段优先级如下：

### 第一优先级
- 继续增强发送观察面与排障能力
- 继续收口运行边界
- 保持现有可运行主链稳定

### 第二优先级
- 正式补齐消息采集与会话记录链
- 将群聊/私聊消息记录纳入系统主线

### 第三优先级
- 建立聊天分析与方案反哺最小闭环
- 让客户互动真正进入需求判断与方案优化流程

---

## 6. 非目标与边界提醒

以下内容当前不应误判为项目主目标：

### 6.1 非目标
- 只做演示性质的企微发送样品
- 只做孤立接口联调而不沉淀业务资产
- 只做消息发送，不做消息记录与理解
- 只做运行，不做客户互动分析反哺

### 6.2 边界提醒
- 不应把项目缩窄为“企业微信消息发送项目”
- 不应忽略群聊与私聊中的客户真实输入
- 不应忽略客户聊天内容对方案和需求判断的修正作用

---

## 7. 当前完成度口径

### 按“企微接入与真实发送能力”口径
- 完成度较高，已具备可运行基础

### 按“可维护的企微触达系统”口径
- 已有关键基础，但仍需增强运行与排障能力

### 按“客户互动记录、分析、反哺的业务系统”口径
- 仍处于能力缺口显著阶段，需尽快补齐

结论：
- 企微项目当前不是从 0 开始
- 但如果不补“记录—分析—反哺”链条，项目仍未达到真实业务预期

---

## 8. V2 版核心结论

企微项目从本版本开始，正式定义为：

> 一个以企业微信为入口，面向客户互动、需求理解、方案优化与业务推进的系统；
> 发送能力只是基础能力之一，不是项目全部。

后续所有规划、设计、开发、验收与汇报，均以本能力边界为准。
