# analyze 停止旧表双写评估 v1

更新时间：2026-03-30

## 1. 目标

本评估用于判断：

> analyze 是否已经可以在某个阶段停止向旧 insight 表双写，只保留新表写入。

当前问题不是“技术上能不能停”，而是“什么时候停风险最低”。

---

## 2. 当前状态

当前 analyze 链路是：
- 旧表写入
- 新表写入

当前 latest insight 链路是：
- 优先读新表
- 查不到回退旧表

所以当前系统已经不是依赖旧表才能工作，而是处于：
- 新表主链已进入运行
- 旧表仍承担兼容职责

---

## 3. 是否可以马上停旧表双写

当前结论：

**不建议立刻停。**

原因：
- 还没明确排查所有可能读取旧表的调用点
- 还没验证 detail 页 / 运营侧是否完全围绕新表运行稳定
- 还没确认历史 conversation 的 insight 是否都能通过新链路覆盖

也就是说，技术上已经接近可以停，但当前还缺“收口确认”。

---

## 4. 何时可以开始考虑停旧表双写

建议至少满足以下条件：

### 条件 1：latest insight 已稳定主读新表
- 新生成的 insight 都能稳定从新表取回
- 回退旧表的触发频率已很低

### 条件 2：前端消费已稳定
- detail 页 insight 区展示稳定
- analyze -> refresh -> latest insight 体验稳定
- D4 摘要等新展示位不再依赖旧表字段

### 条件 3：关键接口验证已通过
- POST analyze 多次验证通过
- GET latest insight 多次验证通过
- 不同 conversation 至少覆盖数条样本

### 条件 4：文档与回滚方案已准备
- 已明确“停旧写”后的回滚方式
- 已明确旧表仅保留历史兼容的口径

---

## 5. 推荐收口方式

### Phase A：继续保持双写（当前）
目标：继续观察新表稳定性。

### Phase B：停止旧表新增能力投资
目标：旧表不再承接任何新字段或新逻辑。

### Phase C：灰度停旧表写
方式：
- 在分支/测试窗口停旧写
- 只写新表
- 保留 latest 旧表回退读

### Phase D：正式停旧表写
条件：
- 灰度验证稳定
- 消费侧不再依赖旧表

---

## 6. 当前最合理的判断

当前还不是“马上停旧写”的时点，
但已经可以明确进入：

> **停止旧表新增投入、准备停旧表双写的过渡阶段。**

也就是说：
- 旧表已经不该继续扩
- 新功能都应围绕新表设计
- 旧表双写应该被视为临时过渡，而不是长期常态

---

## 7. 一句话结论

analyze 当前最合理的策略不是“立刻停旧表双写”，而是：

**继续用新表承接主结构，同时把旧表双写视为短期兼容措施，并开始为停旧写做验证准备。**
