# 企微项目下一阶段优先级与验收标准 v1

## 1. 优先级排序

### P0：真实接入正式化
目标：把当前“验证级 intake”提升为正式可接入链。

#### 范围
- 正式 webhook / callback 接口
- 验签与基础安全校验
- 真实消息解析
- 幂等/重复处理
- 接入日志与失败日志

#### 验收标准
- 至少一条真实企微消息通过正式入口进入系统
- 消息、会话、分析链可继续衔接
- 同一 msgid 重复进入时不会重复写入
- 接收失败日志可追踪

---

### P1：客户识别稳定化
目标：让 customerId -> patientId 映射从“可用”变成“稳定可运营”。

#### 范围
- 映射规则明确
- 冲突处理明确
- 未识别客户可见
- 映射日志可追踪

#### 验收标准
- 至少验证 3 类映射场景：
  1. external_user_id 命中
  2. wecom_user_id 命中
  3. 未匹配命中
- 映射失败对象可查
- 映射成功结果能用于 message / action 主链

---

### P1：业务动作自动化增强
目标：让动作生成逻辑从“单一生成”变成“分级决策”。

#### 范围
- priority 体系
- 动作模板分层
- 去重与升级
- 风险/意向驱动动作差异化

#### 验收标准
- 至少支持 high / medium / low 三档
- 至少支持两类不同动作策略
- 重复动作不重复创建
- 高优先级动作可覆盖/升级低优先级动作

---

### P2：反馈回流闭环
目标：动作执行结果能回流系统，形成业务闭环。

#### 范围
- 动作状态反馈枚举
- 结果回流规则
- 回流后的客户/会话可读视图

#### 验收标准
- 至少一条 outreach 或 task 状态反馈写回成功
- 回流结果在客户/动作查询中可见
- 可区分 pending/done/failed/closed 等状态

---

### P2：运营管理视图准备
目标：为后续产品化和运营使用提供最小指标口径。

#### 范围
- 会话量
- insight 量
- 待跟进动作量
- 高风险客户量
- 动作完成率

#### 验收标准
- 指标口径文档明确
- 至少形成一版 dashboard/list API 草案
- 指标可从现有数据表计算出来
