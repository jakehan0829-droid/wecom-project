# A1 值班巡视清单 v1

## 目标
让 A1 在主Agent 无响应、长时间未更新、或需要常规巡视时，按固定清单完成最小监督，不靠临场判断。

---

## 一、巡视触发条件
满足以下任一条件时，A1 应执行本清单：

1. 距离最近一次有效推进更新超过 30~60 分钟
2. 主Agent 明确出现 token / quota / model unavailable
3. 收到“请巡视 / 请检查 / 请接管”类任务
4. 演示前、关键改动后、服务重启后

---

## 二、固定巡视步骤

### Step 1：读状态板
读取：
- `project/docs/project-progress-board.md`

重点看：
- 当前主线任务
- 当前负责人
- 当前状态
- 最近一次有效进展
- 下一步动作
- 是否需要韩聪介入

### Step 2：跑最小运行检查
执行：
```bash
cd /root/.openclaw/workspace/project
node scripts/runtime-check.js
```

### Step 3：看 PM2 状态与最近错误日志
执行：
```bash
pm2 status chronic-disease-backend
pm2 logs chronic-disease-backend --lines 60 --nostream
```

### Step 4：判断是否存在“假绿”
即使 `/health` 正常，只要满足以下任一条件，也应视为未真正健康：
- 最近错误日志持续出现数据库字段/权限错误
- 关键接口最近联调已失败
- 服务虽然在线，但业务接口已知不可用
- 状态板与实际运行状态明显不一致

### Step 5：输出巡视结论
必须输出：
1. 当前任务
2. 当前进度
3. 当前运行状态（真绿 / 假绿 / 明确异常）
4. 当前阻塞点
5. 建议接管人（A2 / A3 / A6 / 韩聪）
6. 是否需要立即介入

---

## 三、接管分派规则

### 分派给 A2
适用于：
- build 失败
- 新接口逻辑异常
- 字段不一致
- 代码实现和联调结果不一致

### 分派给 A3
适用于：
- PM2 异常
- 进程未在线
- 数据库权限/连接异常
- 服务重启问题
- 环境变量问题

### 分派给 A6
适用于：
- 已接近收口，但状态板/文档/对外说法不一致
- 演示前需复核
- 范围判断不清

### 升级给韩聪
仅限：
- 新 key / 新模型 / 外部真实配置
- 企业微信真实连接
- 高风险动作
- 多个 Agent 无法继续推进

---

## 四、当前巡视重点提醒
当前项目已知容易出现的假绿场景：
1. `/health` 正常，但业务表结构未同步
2. PM2 在线，但最近 error log 持续报数据库字段不存在
3. 状态板显示“已完成”，但新代码路径未重新验证

A1 必须把“业务接口实际可用性”与“进程活着”区分开。
