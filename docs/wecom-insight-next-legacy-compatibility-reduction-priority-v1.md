# 下一阶段最先可收的旧表兼容项优先级 v1

更新时间：2026-03-30

## 1. 目标

在已进入“准备下一阶段旧表兼容收口”前提下，判断：

> 下一阶段最先能收的旧表兼容项是哪一项。

---

## 2. 当前候选项

### 候选 A：latest insight 的旧表回退读
位置：
- `getLatestConversationInsight`

当前状态：
- 主链优先读新表
- 查不到才回退旧表

判断：
- 这是当前最轻的一项旧表兼容逻辑
- 主链已在观察期内表现稳定
- 一旦继续收口，这项最适合作为第一收口目标

### 候选 B：action-feedback 的旧表兼容写
位置：
- `action-feedback.service.ts`

当前状态：
- 旧表仍保留兼容写
- 新表同步写已验证成功

判断：
- 这是下一批可以考虑收的项
- 但因为它属于旁路反馈时间线，略比 latest 回退读敏感

### 候选 C：更大范围旧表兼容逻辑
例如：
- 其他零散旧表查询
- 更进一步的旧表退化策略

判断：
- 不适合作为下一阶段第一刀

---

## 3. 当前推荐排序

推荐顺序：
1. latest insight 的旧表回退读
2. action-feedback 的旧表兼容写
3. 其他旧表兼容逻辑

---

## 4. 为什么 first pick 是 latest insight 回退读

因为它具备几个特点：
- 范围小
- 语义清晰
- 已有新表主读稳定基础
- 收掉后最能明确验证“主链是否已真正不需要旧表兜底”

也就是说：

> 如果下一阶段只收一项，最合理的第一项就是 latest insight 的旧表回退读。

---

## 5. 一句话结论

下一阶段最先可收的旧表兼容项，当前建议是：

**优先收 `getLatestConversationInsight` 中的旧表回退读；它是风险最小、最适合作为下一阶段第一刀的兼容逻辑。**
