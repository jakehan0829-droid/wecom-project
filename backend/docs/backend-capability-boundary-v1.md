# 当前 backend 能力边界说明 v1

## 文档目标
这份文档用于说明：当前企业微信糖尿病慢病管理系统 MVP backend，已经做到什么、真实验证过什么、还没做到什么，以及下一阶段最合理的推进方向。

它不是愿景文档，而是当前真实落地状态说明。

---

## 一、当前结论（一句话）
当前 backend 已从“接口演示骨架”推进到：
**具备最小真实业务联动闭环的可运行、可演示、可巡检后端。**

---

## 二、当前已经具备的能力

### 1. 基础运行能力
- backend 已部署在当前服务器并由 PM2 托管在线
- PostgreSQL 已接入
- 基础健康检查可用
- 演示账号登录可用
- 已具备最小巡检脚本与巡检清单

### 2. 患者基础管理能力
- 创建患者
- 查询患者列表
- 查询患者详情
- 患者标签创建与绑定

### 3. 企微绑定基础能力
- 患者可绑定企微身份
- 已补 patient 存在性校验
- 已补 bindingType 与字段匹配校验
- 已补最小重复绑定控制
- 错误返回已规范化

### 4. 健康记录写入能力
- 血糖记录写入
- 血压记录写入
- 体重记录写入
- 基本数值范围校验

### 5. 医生任务自动生成能力
- 异常血糖可自动生成医生处理任务
- 异常血压可自动生成医生处理任务
- 同患者、同类异常、同日、pending 状态下具备最小防重复能力

### 6. 患者待触达动作能力
- 已具备最小待触达动作模型
- 支持创建待触达动作
- 支持查询待触达动作列表
- 同患者、同动作类型、同日、pending 状态下具备最小防重复能力
- 已补发送结果字段：`sentAt`、`failureReason`
- 已补手动触发发送入口：`POST /api/v1/patient-outreach-actions/:id/send`
- 已验证“发送失败时正确回写 failed 状态与 failureReason”

### 7. dashboard 真实性增强能力
当前 dashboard 已能反映：
- 患者总数
- 已绑定人数
- 高风险患者数
- 今日健康记录总数（血糖 + 血压 + 体重）
- 待处理医生任务数
- 待触达动作数

### 8. 最小业务闭环能力
当前系统已形成以下闭环：
1. 异常健康记录进入系统
2. 自动生成医生任务
3. 自动生成待触达动作
4. dashboard 反映 pending 数变化
5. 医生任务完成后，关联待触达动作自动关闭
6. dashboard pending 数同步下降

## 三、当前已经真实验证过的内容
以下不是“设计上支持”，而是已经实际 build / 重启 / 联调验证通过：

### 已验证通过
- 登录
- 创建患者
- 绑定企微身份
- 异常血糖自动触发医生任务
- 异常血压自动触发医生任务
- 异常记录自动触发待触达动作
- 同日最小防重复逻辑
- dashboard 新统计口径生效
- 医生任务完成后，待触达动作自动关闭
- 手动触发发送入口可用
- 未满足发送条件时，失败状态与 failureReason 可正确回写
- runtime-check 脚本可用

### 已验证通过的核心链路
已验证链路为：
- 登录 → 创建患者 → 绑定 → 写异常记录 → 查看任务 → 查看待触达动作 → 查看 dashboard → 完成任务 → 再看 dashboard / 待触达动作

这条链已经可作为 MVP 演示主线。

---

## 四、当前明确还没做到的边界
这部分很重要，避免误判当前完成度。

### 1. 还没有真实企业微信消息发送成功闭环
当前已具备发送 service 骨架、手动触发发送入口、失败状态回写能力，但尚未真正接通企业微信真实发送 API 并完成成功投递验证。

### 2. 还没有企业微信真实身份打通
当前企微绑定仍属于最小模拟绑定层，不是完整的企微真实身份校验 / 回调 / 同步链路。

### 3. 还没有复杂任务生命周期
当前医生任务与待触达动作属于最小状态流转，不是完整 BPM / workflow。

### 4. 还没有完整权限体系 / 多角色体系
当前仍以 MVP 简化模型为主。

### 5. 还没有完整审计 / 重试 / 消息投递保障机制
例如：
- 消息发送失败重试
- 幂等投递
- 完整操作审计日志

### 6. 还没有前端产品级联调
当前主要完成的是 backend 侧最小真实闭环，不代表产品端已完整可交付。

### 7. 还没有生产级配置管理与正式外部对接
当前更接近“真实可运行 MVP backend”，还不是生产级完整版系统。

---

## 五、当前最适合怎么对外表述
更合适的说法是：

### 推荐表述
- 当前已经完成 MVP backend 的最小真实业务闭环
- backend 已能体现异常记录驱动的后续处理链路
- 已经不是静态接口演示，而是具备联动与状态收口的可运行后端
- 当前可用于内部演示、能力验证、下一阶段接前端与企微真实能力

### 不建议夸大的表述
- 不应说“企微慢病管理系统已经完整上线”
- 不应说“已经完成真实企微消息闭环”
- 不应说“已经具备完整生产能力”

---

## 六、下一阶段最合理的推进方向
按收益排序，下一阶段更合理的方向是：

### 方向 1：真实企微触达接入
把系统内部 `patient_outreach_action` 与真实企微消息发送打通。

### 方向 2：前端 / 演示端联动
让当前 backend 真实闭环可以在界面层被稳定展示。

### 方向 3：任务与触达动作生命周期继续增强
例如：
- 更细的状态
- 处理人
- 备注
- 关闭原因

### 方向 4：自动化监督机制继续增强
把当前主Agent失联接管机制从文档规则升级为更自动的触发机制。

---

## 七、当前总结
当前 backend 的价值不在于“功能很多”，而在于：
- 已从空壳骨架推进到真实联动
- 已有最小闭环
- 已可运行、可演示、可巡检
- 已具备继续接企微真实能力和前端联调的基础

这意味着：
**现在最应该做的，不是再堆更多孤立接口，而是继续把这条真实闭环往外接通。**
